# 企业知识云盘系统开发提示词

## 一、项目名称

企业知识云盘系统 / 企业文件知识库系统 / 企业资料门户系统

---

## 二、项目定位

请开发一个轻量级企业内部文件知识库系统。

这个系统暂时不需要 RAG、向量库、AI 问答功能，核心目标是：

1. 企业内部成员可以通过钉钉身份登录；
2. 系统校验用户是否属于指定钉钉组织；
3. 通过权限后，用户可以访问企业内部的文件资料；
4. 文件以“文件夹”为基本管理单位；
5. 支持 Word、PPT、PDF、图片、Excel 等常见文件上传、管理、预览和搜索；
6. 支持管理员后台进行文件夹、文件、用户、权限配置；
7. 前端风格参考 Cloud / Claude / Notion 类产品：简洁、正式、清爽、留白充足、企业级质感，不要做成传统老旧网盘风格。

最终系统应该更像一个“企业内部资料门户”，而不是传统网盘。

### 1. 前端技术栈

使用：

- 简洁、正式、清爽；

- 文件夹支持“平铺卡片视图”和“列表视图”切换；
- 适合企业内部知识库场景；
- 不要做成传统网盘风格；
- 不要过度花哨；
- 不要强烈渐变；

### 3. 后端技术栈

使用：

- Python
- FastAPI
- SQLite
- SQLAlchemy
- Alembic
- Pydantic
- JWT / Session 均可，但要有清晰的登录态管理

## 四、核心业务逻辑

## 1. 登录与钉钉权限验证

系统必须支持钉钉登录。

请按照企业内部应用的方式设计：

### 登录流程

1. 用户访问系统；
2. 如果未登录，跳转到钉钉登录 / 免登授权页面；
3. 钉钉返回授权 code；
4. 前端将 code 传给后端；
5. 后端使用钉钉 App 的 Client ID、Client Secret 获取企业内部应用 accessToken；
6. 后端通过 code 换取用户身份信息；
7. 后端获取用户的 unionid / userid / name / avatar / mobile / department 信息；
8. 后端校验该用户是否属于指定企业组织；
9. 如果属于组织，创建本系统用户或更新用户信息；
10. 生成系统登录态；
11. 用户进入文件系统首页；
12. 如果不属于组织，拒绝访问，并显示“暂无权限访问该企业知识库”。

### 钉钉配置项

需要在后端 `.env` 中配置：

```env
DINGTALK_CLIENT_ID=
DINGTALK_CLIENT_SECRET=
DINGTALK_CORP_ID=
DINGTALK_AGENT_ID=
DINGTALK_REDIRECT_URI=
JWT_SECRET=
DATABASE_URL=sqlite:///./data/app.db
STORAGE_DIR=./storage
PREVIEW_DIR=./storage/previews
```

### 用户身份字段

系统用户表至少包含：

- id
- ding_userid
- unionid
- name
- avatar
- mobile
- email
- department_id
- department_name
- is_admin
- is_active
- last_login_at
- created_at
- updated_at

### 权限校验原则

必须实现：

- 未登录不能访问任何文件；
- 非指定钉钉组织成员不能访问系统；
- 普通用户只能访问自己有权限的文件夹；
- 管理员可以访问全部文件夹和后台；
- 后端每一个文件访问接口都必须校验权限，不能只靠前端隐藏；
- 文件下载和文件预览必须走后端鉴权接口，不能直接暴露静态文件路径。

## 6. 权限管理

权限分为三层：

### 系统角色

- 超级管理员；
- 管理员；
- 普通用户。

### 文件夹权限

每个文件夹可以配置访问范围：

1. 全公司可见；
2. 指定部门可见；
3. 指定人员可见；
4. 仅管理员可见。

### 文件权限继承

- 文件默认继承所在文件夹权限；
- 如果后续需要，可单独扩展文件级权限；
- 当前 MVP 不强制实现复杂文件级权限。

权限表设计建议：

`folder_permissions` 表：

- id
- folder_id
- permission_type：all / department / user / admin_only
- target_id：department_id 或 user_id
- created_at

## 九、安全要求

必须实现：

- 所有 API 默认需要登录；
- 管理员 API 必须校验 is_admin；
- 文件预览和下载必须后端鉴权；
- 不允许通过 URL 直接访问 storage 文件；
- 文件名要做安全处理，防止路径穿越；
- 上传文件要校验后缀和 MIME 类型；
- JWT_SECRET 必须通过环境变量配置；
- 钉钉 Client Secret 不允许出现在前端代码；
- 管理员操作要写入 AuditLog；
- 普通用户不能访问未授权文件夹；
- 禁用用户不能继续登录或访问系统。

## 十六、技术选型建议

第一阶段推荐使用：

```text
前端：React + TypeScript + Tailwind CSS + shadcn/ui
后端：FastAPI + SQLite + SQLAlchemy
文件：本地磁盘 storage
预览：PDF.js + LibreOffice 转 PDF
权限：钉钉企业内部应用登录 + 后端组织校验
```

SQLite 第一阶段够用，但要注意：

> 文件不要存在 SQLite 里，只存文件路径和元数据。真实文件放在服务器磁盘目录里。

等后续文件量大了，可以再升级：

```text
数据库：SQLite → PostgreSQL
文件存储：本地 storage → MinIO / OSS / S3
搜索：SQLite LIKE → Meilisearch / Elasticsearch
知识问答：后续再接 RAG
```
