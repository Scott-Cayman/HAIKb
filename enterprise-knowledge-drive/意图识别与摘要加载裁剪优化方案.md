# 意图识别与摘要加载裁剪优化方案

## 1. 方案目标

这份方案单独聚焦一个核心问题：

> 在大模型正式推理前，先判断“该去哪里找”，再决定“只加载哪些摘要”，从而提升速度、命中率和稳定性，并缓解上下文压力。

本方案不是替代现有检索链路，而是在现有“摘要检索 + AI 回答”前面增加一层前置路由 **Module**。  
这个 **Module** 的目标是把原本“所有库一起参与检索”的路径，改造成“先选库，再检索，再回答”的路径。

## 2. 你提出的问题，方案如何回答

### 2.1 预设问题不应让模型深度推理

这个判断是对的。

像下面这类问题，本质上不是复杂推理问题，而是“知识分类 + 快速命中”问题：

- “找什么方案”
- “新人入职看什么”
- “设计员资料在哪”
- “有没有某类案例”

这类问题如果直接走完整 RAG + 大模型深推：

- 会慢
- 会浪费上下文
- 会把不相关的库一起带进来
- 可能让模型在无关摘要里分心

因此这类问题应该优先走“预设问题快答”路径。

### 2.2 先做意图识别，再决定加载哪些摘要

这个方向也是对的，而且应成为稳定版的默认方案。

例如：

- 用户问“找什么方案”
- 系统先识别这是“方案库检索意图”
- 只加载方案库相关文件夹摘要 / 文件摘要
- 不加载新人知识库、设计员文件库、创意文件

这样可以同时带来三种收益：

1. 加快速度
2. 提高命中率
3. 缓解上下文压力

### 2.3 如果意图识别本身也影响速度怎么办

这个问题的关键不是“要不要意图识别”，而是“意图识别由谁做、做多重”。

建议采用三级路由：

1. `规则直达`
2. `轻量 AI 选库`
3. `全量兜底`

也就是说，不能所有请求都让大模型先深度判断一遍。  
稳定版应优先用最轻量的方法完成“选库”，只有规则无法判断时，才让 AI 参与。

## 3. 总体方案

### 3.1 新增一层前置路由 Module

建议在现有检索链路前，新增一个独立 **Module**：

- 名称建议：`IntentRouter`
- 核心职责：
  - 识别问题意图
  - 决定应该加载哪些知识库 / 文件夹摘要
  - 决定是否走预设问题快答
  - 决定是否需要进入后续 RAG

建议把它设计成一个清晰的 **Interface**：

输入：

- 用户 query
- 最近一轮或少量相关历史对话
- 当前用户可访问的文件夹范围

输出：

- `intent_type`
- `target_libraries`
- `target_folder_ids`
- `route_mode`
- `should_use_preset_answer`
- `should_enter_rag`

这样后面的检索 **Module** 就不需要自己猜“该搜哪里”，只负责在限定范围内工作。

## 4. 建议的三层执行路径

### 4.1 Path A：预设问题快答

适用场景：

- 已知高频问题
- 回答模板稳定
- 不需要复杂推理
- 更像 FAQ、导航、指路

例如：

- “新人入职看什么”
- “设计员资料在哪”
- “方案一般去哪找”
- “报销流程在哪看”

执行方式：

1. 用户问题进入 `IntentRouter`
2. 命中预设问题规则
3. 直接返回预设答案
4. 同时附带推荐入口或推荐文件夹

返回结果建议包括：

- 简短直接回答
- 推荐进入的库
- 推荐文件夹
- 如有必要，再给 3 到 5 个代表性文件

这一层的目标不是“生成长回答”，而是“快速给方向”。

### 4.2 Path B：轻量 AI 选库

适用场景：

- 规则能判断一部分，但不够稳定
- 用户问题表达比较口语化
- 需要 AI 做“选库”，但不需要它先做完整回答

例如：

- “帮我找适合科技客户的方案”
- “有没有给新人快速上手看的东西”
- “我想找设计侧可以参考的资料”

执行方式：

1. 先不给 AI 全量文件摘要
2. 只给它一级库或文件夹级摘要
3. 让 AI 回答一个轻任务：
   - 这类问题应该优先加载哪些库
   - 哪些库不用加载
4. 再把选中的库作为检索范围，进入正式 RAG

这一层的关键是：

> 让 AI 判断“去哪找”，而不是一开始就让 AI “直接回答”。

### 4.3 Path C：全量兜底

适用场景：

- 路由判断不明确
- 问题跨多个库
- 规则和轻量选库都无法稳定收敛

执行方式：

1. 放宽检索范围
2. 进入混合召回
3. 但仍然要保留上下文预算控制

全量兜底应该存在，但不应成为默认路径。

## 5. 推荐的核心思路：先选库，再加载摘要

### 5.1 不要一开始就加载所有摘要

未来文件越来越多后，最大的风险不是“检索不到”，而是：

- 加载范围太大
- 候选太多
- 重排太慢
- 回答上下文太长

因此稳定版建议把“加载摘要”做成两段式：

1. `粗粒度加载`
   - 只加载文件夹级摘要
   - 或只加载一级知识库摘要
2. `细粒度加载`
   - 只对命中的库加载文件级摘要
   - 只对命中的文件加载证据段

也就是：

```text
先加载库摘要
  -> 选中目标库
  -> 再加载该库中的文件摘要
  -> 再挑该文件中的证据段
```

这是整个方案里最关键的“卡点”。

### 5.2 利用现有 FolderSummary 能力

当前项目里已经有 `FolderSummary` 相关 **Module**，并且一级文件夹、二级文件夹已经有自动总结能力。

这意味着可以直接复用现有能力做“库级路由摘要”：

- 一级文件夹可以视为“知识库入口”
- 二级文件夹可以视为“子类入口”
- `FolderSummary` 可以作为路由判断的摘要源

也就是说，这个方案不是从零开始，而是可以在现有 **Seam** 上加深：

1. 先读取一级文件夹总结
2. 判断应优先进入哪些一级文件夹
3. 必要时再看二级文件夹总结
4. 最后才进入文件摘要检索

## 6. 意图识别的推荐架构

### 6.1 第一层：规则路由

这是最快的一层，应优先命中。

可维护一份：

- 问题模式词典
- 同义词词典
- 库分类词典
- 预设问题表

例如：

| 用户表达 | 目标库 |
| --- | --- |
| 方案、投标、案例、提案 | 方案库 |
| 新人、入职、培训、适应 | 新人知识库 |
| 设计、创意、视觉、海报 | 设计员文件库 |

优点：

- 快
- 稳
- 可控
- 成本几乎为零

缺点：

- 覆盖有限
- 口语化表达容易漏

### 6.2 第二层：轻量 AI 路由

当规则层无法稳定命中时，再启用 AI 路由。

但这里不建议让 AI 看全部文件摘要，而是只看：

- 一级知识库摘要
- 一级文件夹名
- 一级文件夹一句话判断
- 一级文件夹标签

AI 只需输出：

- 应该加载的库
- 不应该加载的库
- 判断理由

这一层非常适合作为“规则兜底 + 智能补充”。

### 6.3 第三层：RAG 检索执行

只有在库被选出来之后，才进入正式文件摘要检索。

这一层再执行：

1. 限定 `file_ids` 或 `folder_ids`
2. 混合召回
3. file-level 聚合
4. 证据选择
5. 回答生成

这样后续链路的上下文规模会天然变小。

## 7. 推荐的稳定版执行流程

建议稳定版默认走这条链路：

```text
用户提问
  -> IntentRouter
      -> 命中预设问题：直接快答
      -> 未命中：规则路由
      -> 规则不确定：轻量 AI 选库
  -> 得到目标知识库 / 文件夹
  -> 只加载目标库的文件摘要
  -> 混合召回
  -> file 级重排
  -> 只带少量高价值证据生成答案
```

这个流程的本质是：

> 把“所有库一起参与回答”改成“先选库，再做回答”。

## 8. 为什么这套方案能提升速度和准确度

### 8.1 对速度的影响

正向收益：

- 少加载无关摘要
- 少做无关重排
- 少给回答模型塞上下文
- 高频问题可直接快答

潜在成本：

- 多了一步路由判断

但只要路由设计成“规则优先、AI 兜底”，整体通常会更快，而不是更慢。

也就是说：

- 不要把意图识别做成一个新的重推理环节
- 要把它做成一个轻量、前置、控范围的裁剪环节

### 8.2 对准确度的影响

当前常见问题是：

- 用户问方案
- 系统把新人知识、设计资料、创意文件也一起带入
- 模型在大范围信息里误判或分心

前置选库后：

- 查询空间更小
- 候选更纯
- 命中率更高
- 重排更容易做准

### 8.3 对上下文压力的影响

这套方案本质上是在回答前增加一道“摘要加载闸门”。

闸门前：

- 只有库摘要

闸门后：

- 只有选中库的文件摘要

最终回答时：

- 只有 topN 文件的核心证据

所以它天然适合未来文件规模增长后的稳定版本。

## 9. 总体落地建议

### Phase 1：先做最小可用版本

目标：

- 不改大链路
- 先把“选库”放到检索前面

建议做法：

1. 建预设问题快答表
2. 建规则路由表
3. 复用 `FolderSummary` 做一级库摘要
4. 先支持“只按一级库裁剪检索范围”

### Phase 2：加入轻量 AI 选库

目标：

- 提升口语化表达和模糊问题的选库准确率

建议做法：

1. 当规则命中弱时，调用轻量 AI 判断应加载哪些库
2. AI 只看库级摘要，不看全量文件摘要
3. 输出选库结果后再进入正式 RAG

### Phase 3：做多层路由和复杂问题支持

目标：

- 面向更复杂问题和更大文件规模

建议做法：

1. 支持一级库 -> 二级文件夹 -> 文件摘要 的逐层下钻
2. 支持跨库问题的多库合并
3. 支持复杂问题的意图拆分后分别选库

## 10. 最终建议

对于你这次提出的需求，我建议把稳定版的默认路线定成：

1. 高频预设问题直接快答，不做深推理。
2. 绝大多数问题先做意图识别，再决定加载哪些库的摘要。
3. 意图识别优先走规则，只有不确定时才让大模型做“选库判断”。
4. 大模型的第一职责不是先回答，而是先决定“该加载哪里”。
5. 正式回答阶段只消费被选中库里的少量高质量摘要和证据。

一句话总结：

> 不是让 AI 一开始看更多，而是让 AI 先决定应该看哪里，再只看它需要看的摘要。

这会比“全库一起检索再让模型自己筛”更适合未来的稳定版本，因为它同时兼顾：

- 速度
- 命中率
- 上下文压力
- 系统稳定性
