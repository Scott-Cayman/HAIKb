# 知识云盘 + AI 回复项目深度分析与优化方案

## 1. 文档目的

本文面向当前“知识云盘 + AI 回复”项目，梳理现有 AI 检索链路、识别核心问题，并给出一套可逐步落地的优化方案。重点覆盖三件事：

1. 梳理当前 AI 检索逻辑与模块职责。
2. 明确“提高文件命中率与最终回答质量”的优化目标与实施路径。
3. 标准化上传后自动生成摘要的流程，包括 Prompt、摘要内容要素和长度限制。

## 2. 项目现状总览

### 2.1 当前链路概览

当前系统已经具备“上传文件 -> 自动生成摘要 -> 摘要入索引 -> Agent 检索摘要 -> 返回推荐文件与答案”的完整闭环。

主链路如下：

```text
用户上传文件
  -> 后台触发摘要生成任务
  -> 仅读取文件前 10 页 / 文本前 100KB / 图片视觉识别
  -> 生成 AI_DOCUMENT_SUMMARY Markdown
  -> 提取结构化字段并写入 document_summaries
  -> 将摘要按标题切片写入 summary_chunks
  -> 向量检索 + 关键词检索混合召回
  -> 简单重排
  -> Agent 组装 evidence 和 related_files
  -> LLM 生成最终回答
```

### 2.2 关键模块索引

| 模块 | 作用 | 当前结论 |
| --- | --- | --- |
| `backend/app/routers/files.py` | 文件上传，上传后触发摘要生成任务 | 上传即自动摘要，设计方向正确 |
| `backend/app/services/document_parser.py` | 提取前 10 页文本或图片路径 | 能保护原文不直接进入 RAG，但覆盖面有限 |
| `backend/app/services/summary_generator.py` | 生成 `AI_DOCUMENT_SUMMARY` | 已有结构化意识，但摘要标准仍偏松散 |
| `backend/app/services/summary_index_service.py` | 串联解析、摘要、存储、索引 | 主流程清晰，但缺少质量门禁 |
| `backend/app/rag/pipelines.py` | 摘要切片入索引 | 只按 Markdown 标题切片，粒度较粗 |
| `backend/app/rag/keyword_store.py` | 关键词召回 | 适合作为兜底，不足以承载语义检索 |
| `backend/app/rag/vector_store_optimized.py` | 本地“伪向量”检索 | 不是外部 embedding 模型，语义能力有限 |
| `backend/app/rag/retriever_optimized.py` | 混合召回与简单重排 | 仅做轻量 boost，无法解决语义偏差 |
| `backend/app/services/agent_service_optimized.py` | 推荐文件评分与最终回答 | AI 重排信息不足，回答与检索目标未完全对齐 |

## 3. 当前 AI 检索逻辑梳理

### 3.1 上传后摘要生成逻辑

#### Step 1. 文件上传触发摘要任务

- 文件上传成功后，系统会立即异步调用 `generate_summary_and_index_task`。
- 这意味着“摘要生成”已经是知识入库的默认流程，而不是人工补充动作。

#### Step 2. 文件内容解析

系统当前采用“只处理有限范围内容”的策略：

- 图片：直接把图片路径交给视觉模型。
- 文本文件：直接读取文本，最大 100KB。
- Office / PDF：统一转换或读取 PDF，并只抽取前 10 页。

该设计的优点是：

- 满足“不能直接检索原文件全文”的约束。
- 成本可控。
- 能把摘要作为唯一知识入口。

当前局限是：

- 如果关键信息不在前 10 页，摘要会天然失真。
- 一些方案类文件常把行业属性、客户画像、技术路线、实施亮点写在中后段，当前摘要可能遗漏。

### 3.2 摘要生成逻辑

系统当前的摘要 Prompt 已经要求输出固定的 Markdown 结构：

- 机器可读元数据
- 文件一句话判断
- 两句话简介
- 标签
- 重要信息摘要
- 可复用价值
- 适合被哪些问题检索到
- 检索关键词扩展
- 解析限制

这说明项目已经从“纯自然语言摘要”进化到“结构化检索摘要”，方向是对的。

但当前仍存在几个问题：

1. 摘要结构有框架，但字段定义不够严格。
2. 缺少明确的字段长度约束和输出质量门禁。
3. 缺少“主题判定、行业归因、适用场景、排除条件”这类高价值语义字段。
4. LLM 摘要失败时会退回规则生成，规则摘要较依赖关键词命中，泛化能力有限。

### 3.3 索引与召回逻辑

当前召回基于摘要而不是原文，具体为：

1. 摘要 Markdown 按二级标题切片。
2. 切片写入 `summary_chunks`。
3. 检索阶段同时执行：
   - 向量检索
   - 关键词检索
4. 合并去重后执行简单重排。

这里的关键点是：当前系统名义上存在“Embedding / Vector 检索”，但实际向量实现是本地哈希分桶词袋相似度，而不是标准语义 embedding 模型。

这会导致两个后果：

1. 对“同义表达、主题归纳、行业抽象”的理解能力较弱。
2. 更像“加强版关键词匹配”，而不是严格意义上的语义召回。

### 3.4 推荐文件与最终回答逻辑

当前 Agent 流程为：

1. 检索器先召回候选摘要切片。
2. `SummaryDocSearchTool` 将命中切片整理成：
   - `evidence`
   - `related_files`
3. Agent 再调用 LLM 对 `related_files` 打分重排。
4. 最后把证据和推荐文件交给 LLM 生成回答。

当前问题的关键在于：AI 重排阶段看到的信息非常少。

当前评分只依赖：

- 文件名
- 一句话判断

没有把以下关键内容送入重排模型：

- 两句话简介
- 标签
- 重要信息摘要
- 命中的具体证据段
- 文件适合回答哪些问题

因此现状会出现一个典型问题：

> 向量或关键词能命中文件，但模型真正推导时发现该文件并不适合回答用户问题；或者文件名不显著，但正文语义很强，现有重排却无法正确抬升。

这正是当前“Embedding 命中率与最终推导效果有偏差”的根源。

## 4. 当前痛点拆解

### 4.1 痛点一：召回目标与回答目标没有统一

当前召回更像“找看起来相关的摘要块”，而业务真正要的是“找最适合回答用户问题的文件”。

两者差异体现在：

- 召回关注局部相似。
- 回答关注文件整体是否真的属于该场景。
- 用户最终关心的是“这份文件能不能用”，而不是“这段摘要像不像”。

### 4.2 痛点二：摘要对检索友好，但对判定友好程度还不够

现有摘要已经有检索关键词和标签，但还不足以支持复杂业务判断，例如：

- “科技类公司方案”
- “适合互联网客户的招商活动方案”
- “偏 toB、大型企业、发布会类项目”

这些问题需要的不只是词匹配，而是：

- 客户类型归因
- 行业归因
- 方案目标归因
- 场景归因
- 排除误判能力

### 4.3 痛点三：当前重排只看文件弱特征

现有 `_ai_score_files()` 的输入过于简化，只看文件名和一句话判断。这样做的代价是：

- 文件名不含目标词时，优质文档难以上浮。
- “一句话判断”如果写得泛，会把多个文件评分拉平。
- LLM 没有足够上下文判断文件是否真正适用于用户问题。

### 4.4 痛点四：摘要切片粒度和索引字段不够细

当前按 Markdown 二级标题切片，这种方式简单，但对于复杂方案类文档有两个问题：

1. 单个 section 可能仍然很大，语义过粗。
2. 切片与结构化字段之间缺少显式映射，难以实现字段级检索。

例如一个更强的系统通常会把以下内容分开建索引：

- 文件总体定位
- 客户画像
- 行业与主题
- 项目目标
- 服务范围
- 可复用模块
- 检索问法样本

### 4.5 痛点五：缺少检索效果评估闭环

当前系统能运行，但还缺少一套稳定的效果评估机制：

- 没有标准测试 query 集合。
- 没有 top1 / top3 命中率统计。
- 没有“召回正确但回答错误”与“召回错误”分层分析。
- 没有线上点击、预览、下载、人工确认的反馈回流。

没有评估闭环，就很难持续优化。

## 5. 优化目标定义

### 5.1 业务目标

系统要从“能搜到相似摘要”升级为“能找出最适合回答问题的文件”。

核心目标如下：

1. 明显提升模糊语义查询下的文件命中率。
2. 提升推荐文件与最终回答之间的一致性。
3. 让文件名未出现查询词时，系统仍能通过摘要语义正确召回和排序。
4. 在文件量持续增长后，仍能控制上下文长度、响应时延和系统稳定性。
5. 形成可持续迭代的摘要标准和检索评估体系。

### 5.2 目标示例

以用户问题“科技类公司方案”为例，系统应具备以下能力：

- 即使文件名没有“科技类公司”字样，也能识别正文或摘要中的行业属性。
- 能区分“科技公司客户方案”和“普通活动方案里偶然提到科技词汇”的差别。
- 能优先推荐真正属于科技类客户、并且内容可复用的方案文件。

### 5.3 可量化指标建议

建议建立以下核心指标：

| 指标 | 定义 | 目标建议 |
| --- | --- | --- |
| Top1 文件命中率 | 用户 query 的最优文件是否排第 1 | 首期提升到 60%+ |
| Top3 文件命中率 | 最优文件是否出现在前 3 | 首期提升到 85%+ |
| 推荐-回答一致率 | 最终答案引用的核心文件是否来自高相关推荐 | 90%+ |
| 模糊语义召回率 | 文件名不含关键词时仍正确召回 | 持续提升 |
| 摘要合格率 | 摘要字段完整、长度合规、可用于索引 | 95%+ |
| P95 响应时延 | 用户发起 AI 检索到返回首个结果的耗时 | 稳定收敛，避免持续劣化 |
| 单次回答上下文大小 | 最终送入回答模型的 token 总量 | 设硬阈值并可监控 |
| 超时降级成功率 | 重排或回答超时时是否能稳定降级返回 | 99%+ |

## 6. 深度优化方案

### 6.1 总体原则

优化建议遵循五个原则：

1. 不改变“只基于摘要检索，不直接检索全文”的总约束。
2. 先增强摘要质量，再增强召回与重排，再优化回答生成。
3. 采用“两阶段召回 + 一阶段重排 + 一阶段回答”的稳定架构。
4. 默认不要依赖超长上下文硬堆结果，而是优先通过召回、压缩、聚合控制上下文预算。
5. 并行拆分只在收益明显时启用，且必须具备超时、重试、降级和结果合并策略。

### 6.2 方案一：把摘要升级成“检索卡 + 判断卡”

当前摘要更像“总结文档”，建议升级为“双层摘要结构”：

#### A. 检索卡

面向召回，强调高密度、短字段、可索引。

建议包含：

- 文件类型
- 客户类型
- 客户行业
- 项目场景
- 核心主题
- 适用问法
- 不适用问法
- 关键词扩展
- 同义表达扩展

#### B. 判断卡

面向重排和回答，强调“是否真的适合”。

建议包含：

- 文件核心用途
- 适用客户画像
- 适用行业
- 适用项目阶段
- 可复用模块
- 关键交付内容
- 为什么推荐
- 为什么可能不适合

这样做的收益是：

- 召回看“检索卡”。
- 重排看“判断卡”。
- 回答看“判断卡 + 证据段”。

这会显著减少“召回命中但回答不佳”的偏差。

### 6.3 方案二：引入真正的语义召回层

当前 `vector_store_optimized.py` 采用的是本地哈希词袋方案，性能轻量，但语义能力明显不足。

建议升级为：

#### Phase 1. 保留现有本地向量作为兜底

- 本地向量仍可保留，用于离线环境或 API 不可用时降级。

#### Phase 2. 接入真实 embedding 模型

可选方案：

- OpenAI 兼容 embedding 接口
- 火山 / 通义等兼容 embedding 服务
- 本地中文 embedding 模型

接入后索引对象建议不是整篇摘要，而是以下多视角文本：

1. 文件一句话判断
2. 两句话简介
3. 标签聚合文本
4. 重要信息摘要
5. 适合被以下问题检索到
6. 检索关键词扩展

建议采用“多向量索引”而不是单向量索引：

- `retrieval_card_vector`
- `judgement_card_vector`
- `question_samples_vector`

这样对于“科技类公司方案”这类抽象问题，召回能力会明显提升。

### 6.4 方案三：把重排从“文件名重排”升级为“证据感知重排”

当前 AI 重排只看文件名和一句话判断，建议升级为以下输入：

- 文件名
- 一句话判断
- 两句话简介
- 客户类型 / 项目类型 / 文件类型
- 行业标签 / 区域标签 / 关键词标签
- 重要信息摘要
- 与 query 最相关的命中摘要段
- 适合被哪些问题检索到

建议输出以下字段：

- `relevance_score`
- `match_reasons`
- `risk_notes`
- `is_recommended`

推荐的重排 Prompt 目标不只是“打分”，而是要求模型判断：

1. 该文件是否真的属于用户要找的主题。
2. 该文件是否可作为复用参考。
3. 该文件只是擦边相关，还是高度匹配。

这样能解决“命中了，但不适合用”的问题。

### 6.5 方案四：先做查询理解，再做召回

用户问题往往不是标准检索词，而是业务语义。建议在检索前增加一层 Query Understanding：

输出结构建议包括：

- 原始 query
- 标准化检索意图
- 客户类型
- 行业类型
- 项目场景
- 文件类型偏好
- 必须满足条件
- 可选加分条件
- 同义词扩展
- 反向排除词

以“科技类公司方案”为例，可扩展为：

- 行业：科技 / 互联网 / 软件 / 数字化 / AI / 平台
- 文件类型：方案 / 策划案 / 投标方案 / 项目方案
- 场景：客户方案 / 服务方案 / 解决方案

这一步不一定每次都调用大模型，也可以采用“规则 + 词典 + LLM 兜底”的混合方案。

### 6.6 方案五：建立字段级混合召回

建议不要只对“大块摘要文本”做召回，而是对结构化字段分别建索引：

| 字段 | 用途 |
| --- | --- |
| `one_line_judgement` | 快速语义定位 |
| `two_sentence_intro` | 概览判断 |
| `client_type` | 客户分类过滤 |
| `project_type` | 场景过滤 |
| `document_type` | 文件类别过滤 |
| `industry_tags` | 行业匹配 |
| `keyword_tags` | 关键词命中 |
| `fit_questions` | 用户问法对齐 |
| `not_fit_questions` | 反向排除 |
| `reusable_value` | 是否适合复用 |

推荐召回公式：

```text
最终召回分 = 
  语义向量分 * 0.45 +
  问法样本匹配分 * 0.20 +
  标签匹配分 * 0.15 +
  关键词命中分 * 0.10 +
  文件类型/客户类型过滤加权 * 0.10
```

这样能够让检索从“单一路径命中”变成“多证据加权命中”。

### 6.7 方案六：回答阶段只消费重排后的高质量证据

当前 `evidence` 是原始召回结果，`related_files` 是重排后的文件列表，两者可能不一致。

建议调整为：

1. 先召回 chunk。
2. 聚合成 file 级候选。
3. 做 file 级 AI 重排。
4. 仅对 topN 文件重新选取最优 chunk 作为最终 evidence。
5. 用“最终 evidence + 最终推荐文件”生成回答。

这样回答阶段看到的上下文会与推荐文件保持一致，减少模型在回答中引用低相关证据。

### 6.8 方案七：加入人工反馈与在线学习闭环

建议采集以下行为数据：

- 用户是否点击推荐文件预览
- 用户是否下载推荐文件
- 用户是否继续追问“不是这个”
- 管理员是否手工修正标签
- 文件是否被加入收藏

这些信号都可以反哺：

- 重排权重
- 标签补全
- 评估集构建
- 高价值文件优先级

### 6.9 方案八：加入上下文预算与并行编排策略

这一部分是稳定版本必须单独考虑的能力，不能默认依赖模型的超长上下文上限。

当前已知现状：

- 当前模型为 DeepSeek V4 Flash，理论上下文上限较高。
- 现阶段文件量和摘要量不大，但单次执行已需要 30 多秒。
- 这说明系统当前的瓶颈不只是“能不能塞进去”，还包括：
  - 上下文装配过大
  - 多轮 LLM 调用累积耗时
  - 模型在长上下文下推理速度下降

因此建议明确一条原则：

> 超长上下文是容量兜底，不应作为默认工作模式。稳定版本应优先采用“预算化上下文 + 分层处理 + 必要时并行拆分”的方案。

#### A. 上下文预算化

每次问答都应有明确预算，而不是把召回到的内容尽量多地塞给模型。

建议拆成三层预算：

1. 查询理解预算
   - 只处理用户原始 query 和必要历史对话。
   - 目标是抽取检索意图，而不是直接回答。
2. 重排预算
   - 只处理候选文件的“判断卡”和少量命中证据。
   - 不引入完整摘要全文。
3. 回答预算
   - 只处理最终 topN 文件和每个文件的最优证据段。
   - 默认控制在一个稳定阈值内，而不是无限扩张。

建议把“单次回答可用上下文”做成配置项，例如：

- `query_understanding_context_budget`
- `rerank_context_budget`
- `answer_context_budget`
- `max_files_for_final_answer`
- `max_chunks_per_file`

#### B. 默认采用“小上下文优先”策略

稳定版本建议优先采用以下策略，而不是直接吃满 1024k：

1. 先召回更多候选，但只给模型看压缩后的判断信息。
2. 先做 file-level 重排，再选少量高价值 evidence。
3. 最终回答只给模型看 top3 到 top5 文件，而不是所有候选文件。
4. 历史对话只保留与当前问题强相关的部分，避免无限累积。

这样做的好处是：

- 延迟更可控。
- 成本更可控。
- 回答质量更稳定。
- 后期文件规模增长时不会线性拖慢。

#### C. 何时使用并行拆分

并行拆分是一个可选增强策略，不应默认全量开启。

建议只在以下场景启用：

1. 候选文件过多，单次重排上下文超过预算。
2. 用户问题本身是复合问题，例如同时问“科技类客户 + 投标方案 + 可复用案例”。
3. 需要对多个候选簇分别判断，再统一汇总。

可采用两类并行方式：

##### 方式 1：候选分组并行重排

做法：

1. 先把候选文件分成多个 batch。
2. 并行让模型对每个 batch 独立判断和打分。
3. 最后再由一个轻量合并步骤输出全局 topN。

适合场景：

- 文件量逐渐增大后，单次 file rerank 过重。

##### 方式 2：意图拆分并行分析

做法：

1. 先把用户 query 拆成多个子意图。
2. 每个子意图独立召回与分析。
3. 最后做结果归并，生成总答案。

适合场景：

- 用户问题本身是多目标、多约束组合。

例如：

```text
找科技类公司方案 + 偏投标场景 + 可复用执行框架
```

可拆为：

- 子任务 1：科技类公司识别
- 子任务 2：投标场景识别
- 子任务 3：可复用执行框架识别

#### D. 并行拆分的工程约束

并行能提升吞吐，但也会带来新的稳定性问题，因此必须有控制策略：

1. 每次请求的并行子任务数设上限。
2. 子任务有超时控制，超时后允许部分结果合并。
3. 子任务失败不影响整体可用性，系统应返回降级结果。
4. 合并阶段必须是轻量步骤，避免“前面并行，最后单点巨慢”。
5. 需要记录每个子任务的耗时、输入规模、输出质量，便于后续调优。

#### E. 建议的稳定版执行策略

对于未来稳定版，建议采用如下默认执行路径：

```text
Query Understanding
  -> 混合召回（多拿候选）
  -> file 级聚合
  -> 预算判断
      -> 若未超预算：单次重排
      -> 若超预算：批次并行重排
  -> 选出 topN 文件
  -> 为每个文件提取最优证据段
  -> 在回答预算内生成最终答案
  -> 若回答阶段超时：降级为“推荐文件 + 简短理由”
```

#### F. 对当前模型选型的建议

DeepSeek V4 Flash 的长上下文能力可以保留为平台优势，但建议在系统设计上这样使用：

1. 正常模式：走预算化小上下文，追求快和稳。
2. 增强模式：在复杂问题或人工触发时，才使用更大的上下文窗口。
3. 兜底模式：在并行任务失败或召回不充分时，允许使用更长上下文做一次补充分析。

这样才能兼顾：

- 效果
- 速度
- 稳定性
- 成本可控

## 7. 摘要生成流程标准化方案

### 7.1 目标

摘要不再只是“给人看的一段总结”，而是标准化的“AI 可检索知识卡”。

它必须同时满足三种用途：

1. 便于向量召回。
2. 便于关键词检索。
3. 便于模型判断文件是否适用。

### 7.2 建议的标准摘要结构

建议升级为以下标准结构：

```markdown
# AI_DOCUMENT_SUMMARY

## 0. 机器可读元数据
- file_id:
- original_name:
- document_type:
- parse_scope:
- parse_pages:
- parse_confidence:

## 1. 文件定位
- 一句话判断:
- 文件主用途:
- 是否偏方案/投标/执行/合同:

## 2. 两句话简介

## 3. 主题归因
- 客户类型:
- 客户行业:
- 项目场景:
- 业务主题:
- 服务类型:

## 4. 重要信息摘要
- 项目背景:
- 服务范围:
- 关键要求:
- 交付内容:
- 亮点模块:
- 资质要求:
- 时间节点:
- 预算金额:

## 5. 适用性判断
- 适合复用到哪些场景:
- 更适合回答哪些问题:
- 不适合回答哪些问题:
- 推荐原因:
- 风险提示:

## 6. 检索增强
- 标签:
- 同义词扩展:
- 典型问法:
- 反向排除词:

## 7. 解析限制
```

### 7.3 摘要 Prompt 设计建议

建议把摘要 Prompt 升级为“结构化抽取 + 检索增强”双目标 Prompt。

#### System Prompt 设计原则

应明确约束模型：

1. 只能基于输入内容生成。
2. 不允许脑补客户类型、行业和金额。
3. 无法确认时必须输出“未识别”。
4. 输出必须严格遵循 Markdown 模板。
5. 重点不是写文章，而是为后续检索、重排、问答服务。

#### User Prompt 设计建议

建议显式要求模型完成以下任务：

1. 判断文件属于什么类型。
2. 提取客户类型、行业、项目场景。
3. 归纳适合被哪些业务问题检索到。
4. 归纳不适合被哪些问题检索到。
5. 输出短句化、字段化内容，避免长篇散文。

#### 推荐 Prompt 样例

```text
你是企业知识库的“摘要建卡器”。

你的目标不是写摘要文章，而是生成一张可用于：
1. 语义召回
2. 文件重排
3. AI 回答

的结构化知识卡。

要求：
1. 只能依据输入文本生成，不能补充原文没有的信息。
2. 对客户行业、项目场景、服务类型做明确判断；无法判断时写“未识别”。
3. 输出短句化、字段化内容，避免空话。
4. “典型问法”至少输出 5 条，“反向排除词”至少输出 3 条。
5. “推荐原因”和“风险提示”必须可用于后续文件重排。
6. 输出必须严格遵循给定 Markdown 模板。
```

### 7.4 摘要内容要素要求

建议将摘要要素分成“必填”“推荐”“可选”三级。

#### 必填字段

- 文件一句话判断
- 两句话简介
- 文件类型
- 客户类型
- 项目场景
- 重要信息摘要
- 典型问法
- 检索关键词扩展
- 解析限制

#### 推荐字段

- 客户行业
- 服务类型
- 可复用价值
- 推荐原因
- 风险提示
- 反向排除词

#### 可选字段

- 区域标签
- 预算金额
- 时间节点
- 资质要求

### 7.5 摘要大小限制建议

建议明确限制，避免摘要过长、噪音过多、索引成本失控。

#### 单文件摘要长度建议

- 总长度：`1200 ~ 2200` 中文字符优先。
- 上限：不超过 `3000` 中文字符。
- 若超限，优先压缩“重要信息摘要”和“检索增强”中的冗余表达。

#### 字段级长度建议

| 字段 | 长度建议 |
| --- | --- |
| 一句话判断 | 40-80 字 |
| 两句话简介 | 80-160 字 |
| 单个标签字段 | 5-30 字 |
| 典型问法 | 每条 15-40 字，5-8 条 |
| 推荐原因 | 50-120 字 |
| 风险提示 | 30-100 字 |

#### 切片建议

摘要入索引时建议按“字段块”切片，而不是只按 Markdown 标题切片：

- 基础定位块
- 主题归因块
- 重要信息块
- 适用性判断块
- 检索增强块

这样更利于检索精度和后续重排。

### 7.6 摘要质量门禁建议

每次摘要生成后，增加自动校验：

1. 必填字段是否齐全。
2. 一句话判断是否为空。
3. 两句话简介是否过于模板化。
4. 典型问法数量是否达标。
5. 总长度是否超限。
6. “未识别”字段占比是否过高。

若校验失败：

- 优先触发一次低温重试。
- 再失败则进入“低质量摘要”状态。
- 后台待管理员补录或二次生成。

## 8. 分阶段落地路线

### Phase 1：稳定版基础建设

建议优先做，目标是先把系统从“能跑”变成“快、稳、可控”：

1. 扩充摘要字段，但控制摘要总长度，避免后续上下文膨胀。
2. 增加 Query Understanding 和 file-level 聚合，减少无效上下文进入模型。
3. 调整 AI 重排输入，从“文件名 + 一句话判断”升级为“摘要关键字段 + 命中证据”。
4. 建立上下文预算机制，明确回答阶段最多带多少文件、多少证据段。
5. 建立超时降级策略，确保慢请求不会拖垮整体体验。

### Phase 2：效果与性能协同优化

建议在 Phase 1 稳定后推进：

1. 接入真实 embedding 模型。
2. 建立字段级向量索引与混合召回权重。
3. 增加预算超限时的批次并行重排能力。
4. 建立更稳定的 evidence 压缩与最终答案上下文装配逻辑。

### Phase 3：规模化与反馈闭环

建议作为中期能力建设，面向文件量增长后的稳定版本：

1. 接入用户点击、下载、收藏、纠错等反馈信号。
2. 建立线上监控看板，重点监控命中率、P95 时延、上下文大小和超时率。
3. 支持标签人工修正后自动重建索引。
4. 建立复杂 query 的意图拆分并行分析能力。
5. 形成“摘要质量 -> 检索质量 -> 重排质量 -> 回答质量 -> 在线反馈”的持续优化闭环。

## 9. 推荐的架构改造图

```text
上传文件
  -> 内容解析器
  -> 摘要建卡器
  -> 摘要质量校验器
  -> 标准化摘要入库
  -> 字段级索引器

用户提问
  -> Query Understanding
  -> 混合召回
  -> File 级聚合
  -> AI 重排
  -> 最终证据选择
  -> 回答生成
  -> 用户反馈回流
```

  - `section_type=retrieval_card`

- 最终答案严格使用最终排序后的文件与证据。

### 10.6 `rag.py` / 管理接口

建议修改方向：

- 增加摘要质量状态查询接口。

- 人工修改标签后自动触发 reindex。

## 11. 风险与注意事项

1. 如果直接引入真实 embedding，但摘要结构不升级，效果提升有限。
2. 如果只升级摘要，不升级重排，仍会出现“召回命中但排序错误”。
3. 如果只升级重排，不建立评估集，就难以证明优化是否真的生效。
4. 如果前 10 页无法覆盖关键信息，后续还需要补充“摘要补样”策略，例如抽取目录页、结尾总结页、命中页。
5. 如果默认依赖超长上下文，即使模型能放下，也可能在时延和稳定性上持续恶化。
6. 如果过早开启高并行，又没有超时和降级机制，系统复杂度会明显上升。

## 12. 最终结论

当前项目的总体架构方向是正确的：

- 已经实现“只基于摘要检索，不直接使用原文全文”。
- 已经具备自动摘要、结构化存储、混合召回、Agent 回答的完整闭环。

但现阶段系统仍停留在“摘要驱动的初级 RAG”阶段，主要瓶颈不在是否有 Embedding，而在于：

1. 摘要结构对“业务判断”的支撑还不够强。
2. 向量层还不是真正的语义 embedding。
3. 缺少对上下文预算、响应速度和稳定性的系统性控制。
4. 重排只看弱特征，导致召回与回答之间存在断层。
5. 缺少评估与反馈闭环。

因此最优改造路径不是单点替换，而是按以下顺序推进：

1. 标准化摘要生成，并控制摘要体积。
2. 建立上下文预算、超时降级和 file-level 聚合机制。
3. 增强字段级召回。
4. 升级 file-level AI 重排，并在必要时支持批次并行。
5. 保证最终回答只消费高质量且受预算控制的证据。
6. 建立评估和反馈闭环。

只要按这个顺序实施，系统就有机会从“能搜摘要”升级成“能真正找到可复用文件并支撑高质量 AI 回复”的企业知识检索系统。
